# Баг-репорты — API микросервиса объявлений

**Окружение:** `https://qa-internship.avito.com`, API v1/v2
**Дата:** 2026-04-05


## Задание 2.1 — Баги API

### BUG-1: Принимается отрицательная цена

- **Приоритет:** P1 — High
- **Эндпоинт:** `POST /api/1/item`
- **Описание:** API принимает объявления с отрицательной ценой (price < 0)
- **Шаги:**
  1. POST /api/1/item с телом `{"sellerID": 555112, "name": "Test", "price": -100, "statistics": {"likes": 1, "viewCount": 1, "contacts": 1}}`
- **Ожидаемый результат:** 400 Bad Request с сообщением о невалидной цене
- **Фактический результат:** 200 OK, объявление создаётся с price=-100
- **Влияние:** Невалидные данные в системе, некорректное отображение цен

---

### BUG-2: Принимается отрицательная статистика

- **Приоритет:** P2 — Medium
- **Эндпоинт:** `POST /api/1/item`
- **Описание:** API принимает объявления с отрицательными значениями likes, viewCount, contacts
- **Шаги:**
  1. POST /api/1/item с телом `{"sellerID": 555115, "name": "Test", "price": 100, "statistics": {"likes": -5, "viewCount": -10, "contacts": -1}}`
- **Ожидаемый результат:** 400 Bad Request
- **Фактический результат:** 200 OK, объявление создаётся с отрицательными значениями статистики
- **Влияние:** Некорректные данные статистики

---

### BUG-3: Нулевые значения трактуются как отсутствующие

- **Приоритет:** P1 — High
- **Эндпоинт:** `POST /api/1/item`
- **Описание:** Значения `0` для полей price, likes, viewCount, contacts воспринимаются как «поле не передано», хотя 0 — валидное числовое значение
- **Шаги:**
  1. POST /api/1/item с `"statistics": {"likes": 0, "viewCount": 1, "contacts": 1}`
- **Ожидаемый результат:** 200 OK (0 — допустимое значение для likes)
- **Фактический результат:** 400 `{"result":{"message":"поле likes обязательно"}}`
- **Причина:** Вероятно, проверка `if !value` вместо `if value == nil` в Go / аналог
- **Влияние:** Невозможно создать объявление с нулевой статистикой (новое объявление без просмотров/лайков)

---

### BUG-4: Ответ POST не содержит полный объект

- **Приоритет:** P2 — Medium
- **Эндпоинт:** `POST /api/1/item`
- **Описание:** Согласно Postman-спецификации, ответ POST должен содержать полный объект объявления (id, sellerId, name, price, statistics, createdAt). Фактически возвращается только `{"status": "Сохранили объявление - <id>"}`
- **Шаги:**
  1. POST /api/1/item с валидными данными
  2. Проверить структуру ответа
- **Ожидаемый результат:** `{"id": "...", "sellerId": 555111, "name": "...", "price": 100, "statistics": {...}, "createdAt": "..."}`
- **Фактический результат:** `{"status": "Сохранили объявление - fd1d9b7d-..."}`
- **Влияние:** Клиенту приходится делать дополнительный GET-запрос для получения данных созданного объявления

---

### BUG-5: Принимется отрицательный sellerId:

- **Приоритет:** P2 — Medium
- **Эндпоинт:** `POST /api/1/item`
- **Описание:** API принимает объявления с отрицательным значением sellerId
- **Шаги:**
  1. POST /api/1/item  `{"sellerID": -1,"name": "iphone","price": 121,"statistics": {"likes": 1542,"viewCount": 8765, "contacts": 231}}`
  2. Проверить структуру ответа
- **Ожидаемый результат:** 400 Bad Request (невалидный ID продавца)
- **Фактический результат:** `{"status": "Сохранили объявление - fd1d9b7d-..."}`
- **Влияние:** Нарушение валидации, доступ к данным с невалидными ID

---

### BUG-6: Несоответствие имён полей (sellerID vs sellerId)

- **Приоритет:** P3 — Low
- **Эндпоинт:** `POST /api/1/item` + `GET /api/1/item/:id`
- **Описание:** POST принимает поле `sellerID` (с заглавными ID), а GET возвращает `sellerId` (с маленькой d)
- **Шаги:**
  1. POST с `"sellerID": 555111`
  2. GET созданного объявления
  3. Сравнить ключ поля
- **Ожидаемый результат:** Одинаковое именование: `sellerId` или `sellerID`
- **Фактический результат:** POST: `sellerID`, GET: `sellerId`
- **Влияние:** Неконсистентность API, затрудняет интеграцию

---

### BUG-7: GET /item/:id возвращает массив вместо объекта

- **Приоритет:** P2 — Medium
- **Эндпоинт:** `GET /api/1/item/:id`
- **Описание:** Запрос одного объявления по ID возвращает JSON-массив с одним элементом, а не единичный объект
- **Шаги:**
  1. Создать объявление
  2. GET /api/1/item/{id}
- **Ожидаемый результат:** `{"id": "...", "name": "...", ...}` — единичный объект
- **Фактический результат:** `[{"id": "...", "name": "...", ...}]` — массив с одним элементом
- **Влияние:** Нарушение REST-контракта, клиентам нужно дополнительно индексировать массив

---

### BUG-8: Отрицательный sellerID принимается в GET

- **Приоритет:** P2 — Medium
- **Эндпоинт:** `GET /api/1/:sellerID/item`
- **Описание:** API принимает отрицательные sellerID и возвращает объявления
- **Шаги:**
  1. GET /api/1/-1/item
- **Ожидаемый результат:** 400 Bad Request (невалидный ID продавца)
- **Фактический результат:** 200 OK с массивом объявлений для sellerId=-1
- **Влияние:** Нарушение валидации, доступ к данным с невалидными ID

---

### BUG-9: Статистика доступна после удаления объявления

- **Приоритет:** P2 — Medium
- **Эндпоинт:** `GET /api/1/statistic/:id` + `DELETE /api/2/item/:id`
- **Описание:** После удаления объявления через DELETE /api/2/item/:id, его статистика по-прежнему доступна через GET /api/1/statistic/:id
- **Шаги:**
  1. Создать объявление
  2. DELETE /api/2/item/{id}
  3. GET /api/1/statistic/{id}
- **Ожидаемый результат:** 404 Not Found
- **Фактический результат:** 200 OK со старыми данными статистики
- **Влияние:** Утечка данных, «осиротевшая» статистика, неконсистентное состояние
